### 詳細分析

#### 1. リアルタイム性能（High）
`SampleDB::find()` は MIDI Note On 時に呼ばれます。もしこれがオーディオ callback スレッド内で実行されている場合、以下の `push_back` はリアルタイム違反（Priority Inversion の原因）になります。
```cpp
// src/sample_db.cpp
candidates_.clear();
if (it != note_index_.end() && it->first - midi_note == best_dist)
    for (size_t idx : it->second) candidates_.push_back(idx); // 容量を超えると realloc 発生
```
`candidates_.reserve(16)` はコンストラクタで行われていますが、ベロシティレイヤーやラウンドロビンの数が多い場合、16を超える可能性があります。

#### 2. FLAC デコーダのバグ・性能リスク（Medium）
未ステージングの変更で `search_end = buf.size()` となっています。
```cpp
// src/flac_decoder.cpp (unstaged)
size_t search_end = buf.size();  // ファイル全体を探索
if (!try_find_encrypted_sync(buf, search_begin, search_end, ...))
```
`read_file_bytes` はファイル全体をメモリに読み込むため、巨大なファイル（例: 数GB）が指定された場合、全バイトに対して4回の XOR ローテーション試行が発生し、起動プロセスがハングしたように見える可能性があります。読み込み前にファイルサイズを制限するか、探索範囲に上限を設けるべきです。

#### 3. PCM 健全性チェック（Low/Good）
`sample_db.cpp` に追加された防御策は、破損したサンプルによるデジタルノイズやクラッシュを防ぐために非常に適切です。特に ±2.0 超の異常値や DC オフセット（無音固着）のチェックは実用的です。

### コードレビュー結果報告

#### 1. リアルタイム安全性の欠如 (重要度: **Critical**)
*   **場所:** `src/sample_db.cpp` の `SampleDB::find()`
*   **詳細:** オーディオスレッド（`AudioOutput` スレッド）内で実行される `SampleDB::find()` において、メンバ変数 `candidates_` に対して `push_back()` が行われています。
*   **リスク:** ベロシティレイヤーやラウンドロビン数が多いサンプルセット（例：16レイヤー以上）を使用した場合、ベクトルの容量（`reserve(16)`）を超えて**ヒープの再確保（realloc）**が発生する可能性があります。これはオーディオの途切れ（ドロップアウト）の直接的な原因となります。
*   **対策:** `candidates_` を固定長配列にするか、オーディオスレッド外で十分に大きく確保されたバッファを使用するように修正が必要です。また、`std::map::lower_bound` のポインタ追跡によるキャッシュミスも、将来的な最適化対象です。

#### 2. オーディオスレッドの優先度未設定 (重要度: **High**)
*   **場所:** `src/main.cpp`
*   **詳細:** `std::thread` でオーディオおよび MIDI スレッドを生成していますが、`pthread_setschedparam` 等によるリアルタイム優先度（`SCHED_FIFO`）の設定が行われていません。
*   **リスク:** Raspberry Pi 5 のようなマルチタスク環境では、OS の他のプロセスによってオーディオスレッドがプリエンプト（中断）され、バッファアンダーランが発生しやすくなります。

#### 3. FLAC デコーダの堅牢性と起動性能 (重要度: **Medium**)
*   **場所:** `src/flac_decoder.cpp` (未ステージングの変更)
*   **詳細:** 暗号化 sync が見つからない場合、`search_end = buf.size()` としてファイル全体を走査するよう変更されています。
*   **リスク:** 巨大な（数GB規模など）不正なファイルを読み込ませた場合、デコード時に $O(4N)$ の走査が発生し、アプリケーションが長時間ハングアップ（起動不能）状態になります。
*   **対策:** 探索範囲に合理的な上限（例: 500KB）を設けるか、読み込み前にファイルサイズを制限すべきです。

#### 4. セキュリティ: パストラバーサルガード (重要度: **Medium**)
*   **場所:** `src/sample_db.cpp`
*   **詳細:** `std::filesystem::relative` を用いたガードが実装されていますが、Issue #20 で指摘されている通り、シンボリックリンク等が絡む場合にバイパスされる可能性が残ります。
*   **対策:** `weakly_canonical` を適用したパスが、ベースディレクトリのパスで前方一致するかを文字列レベルで追加検証する等の強化が望まれます。

#### 5. PCM 健全性チェックの実装 (評価: **Good**)
*   **詳細:** `sample_db.cpp` に実装された 4 段階の防御（デコード例外捕捉、極端に短いサンプルの除外、±2.0超の異常値検知、DC オフセット検知）は、破損したサンプルによるスピーカーの損傷や不快なノイズを未然に防ぐための非常に優れた実装です。
